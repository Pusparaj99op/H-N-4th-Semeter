Practical No. 03 – Study of Basic Networking Commands

Aim

To study and demonstrate important built-in networking commands in Windows OS and understand their purpose in network diagnostics and configuration.

Theory

Networking commands are built-in utilities that allow users and administrators to configure network settings, test connectivity, diagnose issues, and gather network information. Key Windows networking commands are:

1. ipconfig – Displays the IP configuration for all network adapters. Essential for finding the IP address, subnet mask, default gateway, MAC address, and DNS servers.

CommandPurpose
ipconfigShows IP address, Subnet Mask, Default Gateway for each adapter
ipconfig /allShows complete details including MAC address, DHCP status, DNS servers
ipconfig /releaseReleases the currently assigned DHCP IP address
ipconfig /renewRequests a new IP address from the DHCP server
ipconfig /flushdnsClears the DNS resolver cache

2. ping – Tests connectivity between two hosts by sending ICMP (Internet Control Message Protocol) Echo Request packets and waiting for Echo Reply. Reports round-trip time (RTT) and packet loss.

CommandPurpose
ping 127.0.0.1Loopback test – verifies TCP/IP stack is working on the local machine
ping 192.168.1.1Pings the default gateway to test local network connectivity
ping www.google.comTests internet connectivity and DNS resolution
ping -t [IP]Sends continuous ping packets until stopped (Ctrl+C)
ping -n 10 [IP]Sends exactly 10 ICMP packets

3. tracert (Trace Route) – Displays the path (hops) that packets take to reach a destination host. Shows the IP of each router along the route and time taken at each hop. Useful to identify where a network failure or slowdown occurs.

4. netstat – Displays active TCP/IP connections, listening ports, and protocol statistics.

CommandPurpose
netstatLists currently active connections
netstat -aShows all connections and listening ports
netstat -nDisplays addresses and port numbers in numerical format
netstat -rDisplays the IP routing table

5. nslookup – Queries DNS (Domain Name System) servers to resolve domain names to IP addresses, or reverse-lookup IPs to domain names. Useful for troubleshooting DNS issues.

6. arp – Displays and modifies the ARP (Address Resolution Protocol) cache, which stores mappings of IP addresses to MAC (physical) addresses.

CommandPurpose
arp -aDisplays all current ARP cache entries (IP to MAC mappings)
arp -d [IP]Deletes a specific ARP cache entry

7. hostname – Displays the computer's NetBIOS or DNS hostname.

8. net view – Lists computers and shared resources visible on the network.

Apparatus / Requirements

  • Computer System with Windows OS (Windows 10 or later)
  • Command Prompt (cmd.exe) with Administrator privileges
  • Active LAN / Wi-Fi network connection
  • Internet connection (for testing ping to external hosts)

Procedure

  1. Open Command Prompt: Press Win + R, type cmd, and press Enter. (For admin: right-click cmd → "Run as administrator".)
  2. ipconfig:
    • Type ipconfig and press Enter. Record the IPv4 Address, Subnet Mask, and Default Gateway.
    • Type ipconfig /all. Note the Physical (MAC) Address and DNS Servers.
  3. ping:
    • Type ping 127.0.0.1 – loopback test. Verify 0% packet loss.
    • Ping your default gateway: e.g., ping 192.168.1.1.
    • Ping an external server: ping www.google.com. Note TTL and time values.
  4. tracert:
    • Type tracert www.google.com. Observe each hop and its latency. Stop with Ctrl+C if needed.
  5. netstat:
    • Type netstat -a and note active TCP/UDP connections and listening ports.
  6. nslookup:
    • Type nslookup www.google.com. Record the IP address(es) returned by DNS.
  7. arp:
    • Type arp -a. Note the MAC addresses of nearby devices in the ARP cache.
  8. Type hostname and note the computer name.
  9. Document all command outputs with screenshots or written notes in the journal.

Result

Basic networking commands — ipconfig, ping, tracert, netstat, nslookup, arp, and hostname — were successfully executed and their outputs were analyzed. Network connectivity was verified at multiple levels: loopback (127.0.0.1), local gateway, and external internet (www.google.com). DNS resolution, ARP cache, and active port status were also examined.
